# RL SFT-v9 KL 200-step 监控记录

## 目标

从 `models/TimeThinker-4B-SFT-v9-10k-canonical` 启动一条 200-step GRPO 训练，修复旧 run 中 KL 实际未启用的问题，并在训练过程中持续检查 reward、固定验证集、KL、entropy、梯度、长度、显存和吞吐。

## 初始配置

- 配置：`config/rl/qwen3_rl_sftv9_kl200.yaml`
- 输出：`models/TimeThinker-4B-RL-sftv9-kl01-len02-bs16-200`
- `disable_kl: false`、`use_kl_loss: true`、`kl_coef: 0.01`
- `len_reward: 0.02`（旧 run 为 0.1）
- `format_weight: 0.05`（旧 run 为 0.2）
- 固定验证使用 greedy decoding，`val_before_train: true`、`val_freq: 25`
- `max_steps: 200`、`save_freq: 50`、保留最近 2 个可续训 checkpoint

## 自动干预规则

- 启动阶段 OOM：降低 rollout GPU memory utilization 或继续增加 reference offload。
- `actor/kl_loss` 缺失：视为 KL wiring 失败，停止并修复配置。
- KL 快速增大且验证下降：提高 `kl_coef`；KL 近零且 entropy 快速坍缩：降低学习率或提高 `kl_coef`。
- 梯度持续非有限或远超裁剪阈值：停止并降低学习率。
- 固定验证连续两次明显下降：从最近 checkpoint 调整后续训练，避免盲目跑满。
- 每次实际配置修改都追加到下方记录，不覆盖历史判断。

## 指标口径与量化阈值

- `reward/*` 汇总的是每轮所有生成候选，包含随后被 online filtering 丢弃或截掉的组，因此不要求单调增长；模型是否进步以每 25 step 的固定 `val/accuracy_reward` 为主。
- step-0 `val/accuracy_reward=0.672995`。达到 0.683/0.693 分别视为约 +1/+2 个百分点；低于 0.653 且连续两次才调整，单次低于 0.633 视为强警报。
- `val/format_reward` 基线 0.980469；低于 0.96 或下降至少 2 个百分点警报。回答截断率持续高于 5% 警告，高于 10% 调整长度策略。
- `actor/kl_loss` 是未乘 0.01 系数且单步统计偏噪：持续高于 0.05 警告；高于 0.1 且验证连续下降时提高 KL。到 step 25 仍低于 1e-4 且验证及策略指标完全不动时检查更新强度。
- `actor/grad_norm` 是裁剪前范数，当前 `max_grad_norm=5`；高于 5 的比例超过 20%、任一点高于 10 或出现非有限值时停查。
- `actor/entropy_loss` 是采样 token NLL，只看相对趋势；较 step 1 的 0.5585 下降超过 20%，或低于 0.35 且验证不升时警告。
- `reward/length_bonus / 0.02` 可近似看作候选命中 180–372 token 长度奖励的比例；持续 20 step 低于 10% 或高于 80% 说明区间失效或发生长度扎堆。
- 日志中每 step 连续生成至少 3 批才凑齐 16 个过滤后 prompt 组时记录效率警告，至少 5 批视为严重；结合 advantage 极值判断组内方差是否塌缩。

## 为什么 reward 没有形成稳定收益

1. 日志里的 `reward/*` 是过滤前所有生成候选的汇总，包含随后被丢弃的 prompt 组；训练数据又是随机批次，因此它本来就不应单调上涨，也不能直接和固定验证 reward 比较。
2. 当前 online filtering 主要按每组平均 accuracy 过滤，没有要求同一 prompt 的多个 rollout 之间存在足够 reward 方差。对 open-ended、OCR 等任务，可能保留“整组分数几乎相同”的样本；GRPO 归一化后这些组几乎不给有效策略梯度，却仍消耗大量生成算力。
3. open-ended 主要依赖单参考答案的 ROUGE。语义正确但措辞不同的回答容易被低估，错误但词面重合的回答又可能得分偏高，reward 与真实质量不完全对齐。
4. OCR 当前使用按空格切词的 WER；中文和其他无空格文本会退化为非常粗糙的比较。数据类型大小写不统一还可能导致 OCR 专属 prompt 模板未命中。
5. 数据字段整体可用，只有极少数空问题文本，这不是主因。主要瓶颈更像是 reward 可辨别性和过滤后的有效训练信号，而不是 KL、显存或训练数值稳定性。

## 下一轮优化优先级

1. **先修 reward 和过滤，再增加步数。** 对每个 prompt 计算最终 reward 的组内标准差或极差，只保留 `std > ε` 且分数不过饱和的组；同时按任务类型记录保留率、reward 方差和有效 advantage 比例。
2. **按任务类型改 reward。** OCR 对中文/无空格文本使用字符级 CER，对英文保留词级 WER，并统一大小写、空白和标点；open-ended 使用归一化 token F1/ROUGE-L 与语义相似度组合，条件允许时增加多参考答案或离线 judge；选择题、回归任务继续使用可解释的精确匹配或容差规则。
3. **只记录真正参与更新的数据指标。** 将过滤前候选、过滤后候选和最终 actor batch 分开上报，避免把采样难度变化误判为学习曲线。
4. **对齐验证和保存频率。** 下一轮使用 `val_freq=25`、`save_freq=25`，并按 `val/accuracy_reward` 保存 best；至少比较 2–3 个随机种子，确认提升超过噪声。
5. **最后再调更新强度。** reward 修复后若 KL 仍长期低于约 0.02 且固定验证平台，可做 `lr=1e-6/2e-6` 与 `kl_coef=0.005/0.01` 的小规模对照；若验证随 KL 持续上升而下降，再提高 KL。当前结果不支持仅靠延长训练或盲目提高 KL 来解决问题。
